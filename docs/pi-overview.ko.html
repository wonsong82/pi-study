Pi 코딩 에이전트

pi의 작동 방식과 커스터마이징 방법을 기억하기 위한 탐색형 개요, 소개, 자기 학습 노트입니다.

터미널 코딩 에이전트 확장 스킬 프롬프트 템플릿 테마 패키지

일치하는 섹션이 없습니다. 다른 검색어를 입력해 보세요.

pi란?

pi는 최소주의 터미널 코딩 하네스입니다. LLM을 작업 디렉터리에 연결하고 모델에게 read, write, edit, bash 같은 도구를 제공합니다.

모든 워크플로를 내장 기능으로 제공하기보다, 확장, 스킬, 프롬프트 템플릿, 테마, 패키지, 설정, 컨텍스트 파일로 사용자가 자신의 방식에 맞게 조정하는 구조입니다.

기억 모델

pi는 하네스, LLM은 추론 엔진, 도구는 손, 확장/스킬/템플릿은 pi에게 내 워크플로를 가르치는 방법입니다.
항목이렇게 생각하기

Extension런타임 동작: pi가 할 수 있는 일을 바꾸는 TypeScript 코드.

Built-in and active tools

pi에는 registered tools와 active tools의 구분이 있습니다.

개념의미
Registered toolspi가 알고 있는 도구입니다. built-in일 수도 있고 extension이 제공한 도구일 수도 있습니다.
Active tools현재 모델에게 callable tool schema로 전달되는 도구입니다.

기본적으로 pi는 보통 아래 coding tools를 active 상태로 시작합니다.

read, bash, edit, write

하지만 pi에는 mode/extension이 활성화할 수 있는 추가 built-in read-only exploration tools도 있습니다.

grep, find, ls
ToolType하는 일
readDefault active파일 내용을 읽습니다. 텍스트와 이미지를 지원하며, 큰 텍스트 출력은 잘리고 offset/limit로 이어서 읽을 수 있습니다.
bashDefault active현재 작업 디렉터리에서 shell command를 실행합니다. 테스트, git, 검색, 스크립트, 프로젝트 명령에 유용합니다.
editDefault active정확한 텍스트 치환으로 기존 파일을 정밀하게 수정합니다. 작은 surgical modification에 적합합니다.
writeDefault active파일을 생성하거나 전체 내용을 덮어씁니다. 새 파일 또는 complete rewrite에 적합합니다.
grepBuilt-in, not always active파일 내용에서 패턴을 검색합니다. bash grep보다 구조적이며 project ignore behavior를 따릅니다.
findBuilt-in, not always activeglob/pattern으로 파일을 찾습니다. shell command 없이 codebase를 탐색할 때 유용합니다.
lsBuilt-in, not always active디렉터리 내용을 나열합니다. 안전한 read-only exploration에 유용합니다.
예: plan-mode extension은 기본 coding mode가 보통 read, bash, edit, write만 active로 시작하더라도 grep, find, ls를 활성화합니다.

기본 시스템 프롬프트

아래는 pi의 out-of-the-box 기본 시스템 프롬프트 형태입니다. 실제 effective prompt는 날짜, 작업 디렉터리, 설치 경로, 활성 도구, 컨텍스트 파일, 스킬, 확장에 따라 달라질 수 있습니다.

이 프로젝트/세션 기준 정확한 기본 프롬프트 
You are an expert coding assistant operating inside pi, a coding agent harness. You help users by reading files, executing commands, editing code, and writing new files.

Available tools:
- read: Read file contents
- bash: Execute bash commands (ls, grep, find, etc.)
- edit: Make precise file edits with exact text replacement, including multiple disjoint edits in one call
- write: Create or overwrite files

In addition to the tools above, you may have access to other custom tools depending on the project.

Guidelines:
- Use bash for file operations like ls, rg, find
- Use read to examine files instead of cat or sed.
- Use edit for precise changes (edits[].oldText must match exactly)
- When changing multiple separate locations in one file, use one edit call with multiple entries in edits[] instead of multiple edit calls
- Each edits[].oldText is matched against the original file, not after earlier edits are applied. Do not emit overlapping or nested edits. Merge nearby changes into one edit.
- Keep edits[].oldText as small as possible while still being unique in the file. Do not pad with large unchanged regions.
- Use write only for new files or complete rewrites.
- Be concise in your responses
- Show file paths clearly when working with files

프롬프트 생성 흐름

pi는 사용자가 입력한 텍스트만 모델에게 보내지 않습니다. 실제 요청은 세 가지 stream으로 조립됩니다.

Stream포함 내용예시
A. Instruction streameffective system/developer prompt.기본 prompt 또는 SYSTEM.md, APPEND_SYSTEM.md, AGENTS.md, available skills list, date/cwd, extension prompt changes.
B. Conversation stream모델에게 전달되는 messages.이전 user/assistant messages, 확장된 현재 user message, extension custom messages, 이전 tool calls/results.
C. Tool interface호출 가능한 tool definitions/schemas.read, bash, edit, write, extension tools. 이것도 input token을 소비합니다.
핵심 구분: AGENTS.md는 system prompt context에 들어가고, prompt template은 사용자 메시지를 바꾸며, skill은 system prompt에 목록으로 소개되고 필요 시 사용자 context로 로드되며, extension은 거의 모든 단계를 수정할 수 있습니다.
1–4. 리소스 로딩과 base system prompt

시작 또는 /reload 시 pi는 설정, 패키지, 확장, 스킬, 프롬프트 템플릿, 테마, 컨텍스트 파일, SYSTEM.md, APPEND_SYSTEM.md를 발견하고 나중에 사용할 수 있도록 메모리에 유지합니다.

프롬프트 템플릿은 여기서 발견되어 /review 같은 slash command로 바로 사용할 수 있습니다. 스킬도 발견되며, metadata는 system prompt에 소개될 수 있고 full content는 보통 명시적으로 호출하거나 읽을 때 로드됩니다.

base system prompt는 둘 중 하나로 시작합니다.

built-in default prompt
OR
SYSTEM.md

SYSTEM.md가 있으면 내장 default prompt를 완전히 대체합니다. 그 후 pi는 아래 순서로 내용을 append합니다.

1. built-in default prompt OR SYSTEM.md
2. APPEND_SYSTEM.md / --append-system-prompt
3. AGENTS.md / CLAUDE.md project context
4. available skills list, if read tool is active
5. Current date
6. Current working directory

Append는 base prompt를 대체하지 않고 system-level instruction을 뒤에 추가한다는 뜻입니다. 추가 전역/프로젝트 규칙은 APPEND_SYSTEM.md, 저장소/프로젝트 가이드는 AGENTS.md에 두는 식으로 생각하면 됩니다.

5–7. 사용자 입력 확장과 extension hooks

사용자 입력은 모델 호출 전 처리됩니다.

  1. /plan 같은 extension command라면 확장이 직접 처리하고 모델 호출이 없을 수 있습니다.
  2. input hook이 raw text/images를 변환하거나 처리할 수 있습니다.
  3. /skill:name args는 full skill body와 args를 포함한 user message로 확장됩니다.
  4. /review foo 같은 prompt template은 일반 user-message text로 확장됩니다.
  5. before_agent_start hook은 custom message를 추가하거나 이번 턴의 effective system prompt를 수정할 수 있습니다.

중요: /skill과 prompt template은 base system prompt가 아니라 conversation/user-message stream을 수정합니다.

8. 최종 모델 요청

최종 요청은 하나의 평면 문자열이 아니라 구조화된 요청입니다. 개념적으로는 다음과 같습니다.

{
  system: "effective system prompt",
  messages: [
    previous user messages,
    previous assistant messages,
    previous tool calls,
    previous tool results,
    current expanded user message,
    extension-injected custom messages
  ],
  tools: [
    read schema,
    bash schema,
    edit schema,
    write schema,
    extension tool schemas
  ]
}

system prompt, conversation history, 현재 user message, custom messages, active tool schemas, 이전 tool results가 모두 input token에 기여합니다. 대화가 너무 길어지면 session behavior에 따라 오래된 history가 compact/summarize될 수 있습니다.

9. 도구 루프

모델이 작업공간을 읽거나 수정하고 싶으면 tool call을 냅니다. pi는 이를 실행하고 결과를 다음 모델 호출에 다시 포함합니다.

model asks for read(package.json)
→ pi emits tool_call hooks
→ extension may allow/block/modify
→ pi executes read
→ tool result is appended to conversation
→ model is called again with the result
→ repeat until assistant finishes

확장은 이 루프를 가로채 위험한 bash 명령을 막거나, plan mode에서 도구를 제한하거나, 확인을 요청하거나, 사용 기록을 남기거나, input을 수정하거나, 렌더링을 바꿀 수 있습니다.

각 커스터마이징이 token stream에 들어가는 위치
소스들어가는 위치
AGENTS.md / CLAUDE.mdsystem prompt의 project context로 append됩니다.
SYSTEM.md내장 기본 system prompt를 대체합니다.
APPEND_SYSTEM.mdbase prompt 뒤에 system prompt로 append됩니다.
Prompt template모델 호출 전 user message를 확장합니다.
/skill:name전체 skill을 읽어 user message에 주입합니다.

Tool schemas모델이 호출 가능한 도구 정의로 별도 전달되며 input token을 소비합니다.

인터랙티브 사용 기본

  • 터미널 에디터에 일반 요청을 입력합니다.
  • @로 프로젝트 파일을 참조합니다.
  • /로 명령, 프롬프트 템플릿, 스킬 명령을 엽니다.
  • !command는 셸 명령을 실행하고 결과를 LLM에게 보냅니다.
  • !!command는 셸 명령을 실행하지만 결과를 LLM에게 보내지 않습니다.
  • 자주 쓰는 명령: /model, /settings, /resume, /new.

Extensions

확장은 파일/폴더 규칙에 따라 자동 발견됩니다.

위치범위
~/.pi/agent/extensions/*.ts전역 단일 파일 확장
~/.pi/agent/extensions/*/index.ts전역 폴더 확장
.pi/extensions/*.ts프로젝트 로컬 단일 파일 확장
.pi/extensions/*/index.ts프로젝트 로컬 폴더 확장
기본 확장 형태 
import type { ExtensionAPI } from "@earendil-works/pi-coding-agent";

export default function myExtension(pi: ExtensionAPI): void {
  pi.registerCommand("hello", {
    description: "Say hello",
    handler: async (_args, ctx) => {
      ctx.ui.notify("Hello from my extension", "info");
    },
  });
}
확장이 할 수 있는 일
  • pi.registerCommand()로 슬래시 명령 등록.
  • pi.registerTool()로 모델이 호출할 수 있는 도구 등록.
  • pi.registerFlag()로 CLI 플래그 등록.
  • pi.on(...)으로 라이프사이클/세션/모델/도구 이벤트 구독.
  • 도구 호출 차단 또는 수정.
  • 상태 표시, 위젯, 오버레이, 커스텀 에디터, 렌더러 추가.
  • 세션 상태 저장.
보안 주의: 확장은 코드이며 시스템 권한으로 실행됩니다. 신뢰할 수 있는 확장만 설치하거나 복사하세요.

Extension events

확장은 pi.on(eventName, handler)로 pi lifecycle event를 들을 수 있습니다. 이벤트를 통해 resources, session behavior, prompt/message context, tool calls, UI state, provider request를 관찰하거나 수정할 수 있습니다.

pi.on("tool_call", async (event, ctx) => {
  if (event.toolName === "bash") {
    // tool call을 inspect, mutate, block 할 수 있음
  }
});

pi.on("tool_call", async (event) => {

Plan mode 예제 학습

프로젝트 로컬 경로:

.pi/extensions/plan-mode/index.ts
.pi/extensions/plan-mode/utils.ts
.pi/extensions/plan-mode/README.md
하는 일
  1. Plan mode: 읽기 전용 탐색을 위해 도구를 제한합니다.
  2. Bash allowlist: ls, grep, find, git diff 같은 안전한 명령만 허용합니다.
  3. Plan extraction: Plan: 섹션의 번호 목록에서 단계를 추출합니다.
  4. Execution mode: 일반 도구 접근을 복원하고 진행 상황을 추적합니다.
  5. Completion markers: assistant가 [DONE:n]으로 완료 단계를 표시합니다.
  6. UI progress: footer/status widget에 진행 상황을 보여줍니다.

명령: /plan, /todos, Ctrl+Alt+P.

학습 중 발견한 점과 gotchas
발견왜 중요한가
/plan은 mode를 toggle하고 todos를 reset합니다예제는 plan mode를 toggle할 때마다 todoItems를 비웁니다. 단순한 표시 변경이 아닙니다.
setActiveTools()는 active tools를 교체합니다plan mode는 hard-coded tool list를 사용합니다. 다른 extension이 활성화한 tools와 merge하지 않습니다.
다른 extension tools가 비활성화될 수 있습니다다른 extension이 custom tools를 등록/활성화했더라도 plan mode toggle로 모델에게 보이지 않게 될 수 있습니다.
NORMAL_MODE_TOOLS도 hard-coded입니다plan mode를 끄면 사용자의 이전 active tool set이 아니라 read, bash, edit, write만 복원합니다.
grep, find, ls는 built-in이지만 default active는 아닙니다pi가 이 도구들을 알고 있으므로 normal coding mode가 보통 read, bash, edit, write로 시작해도 plan mode가 활성화할 수 있습니다.
questionnaire는 built-in이 아닙니다별도 example extension에서 제공됩니다. 해당 extension이 로드되지 않았다면 setActiveTools()에서 무시됩니다.
Registered tools vs active toolsplan mode는 active tools를 바꿉니다. tools를 생성, 삭제, unregister하는 것은 아닙니다.

더 견고한 plan-mode 구현이라면 plan mode를 켜기 전에 pi.getActiveTools()를 저장하고, 끌 때 그 목록을 정확히 복원해야 합니다.
