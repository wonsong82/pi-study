1. What is pi?

pi is a minimal terminal coding harness. It connects an LLM to a coding workspace and gives the model tools such as read, write, edit, and bash.

Instead of shipping every workflow as a built-in feature, pi expects users to adapt it with extensions, skills, prompt templates, themes, packages, settings, and context files.

4. Default system prompt

This is pi's out-of-the-box default system prompt shape. The date, working directory, installed pi path, active tools, loaded context files, skills, and extensions can change the effective prompt at runtime.

Exact default prompt for this project/session 
You are an expert coding assistant operating inside pi, a coding agent harness. You help users by reading files, executing commands, editing code, and writing new files.

Available tools:
- read: Read file contents
- bash: Execute bash commands (ls, grep, find, etc.)
- edit: Make precise file edits with exact text replacement, including multiple disjoint edits in one call
- write: Create or overwrite files

In addition to the tools above, you may have access to other custom tools depending on the project.

Guidelines:
- Use bash for file operations like ls, rg, find
- Use read to examine files instead of cat or sed.
- Use edit for precise changes (edits[].oldText must match exactly)
- When changing multiple separate locations in one file, use one edit call with multiple entries in edits[] instead of multiple edit calls
- Each edits[].oldText is matched against the original file, not after earlier edits are applied. Do not emit overlapping or nested edits. Merge nearby changes into one edit.
- Keep edits[].oldText as small as possible while still being unique in the file. Do not pad with large unchanged regions.
- Use write only for new files or complete rewrites.
- Be concise in your responses
- Show file paths clearly when working with files

5. Prompt generation flow

Pi does not send only your typed text to the model. The model input is assembled from the effective system prompt, conversation history, your expanded user message, extension-injected messages, tool schemas, and tool results.

Key distinction: AGENTS.md affects system prompt context, prompt templates affect your user message, skills are advertised in the system prompt and can be loaded into user context, and extensions can modify almost every stage.
Step-by-step details
  1. Startup/reload: pi loads settings, packages, extensions, skills, prompt templates, themes, context files, SYSTEM.md, and APPEND_SYSTEM.md.
  2. Extensions register capabilities: tools, commands, flags, input hooks, tool hooks, and before_agent_start hooks.
  3. Active tools are selected: usually read, bash, edit, write, but extensions like plan-mode can change them.
  4. Base system prompt is built: default prompt or SYSTEM.md, plus appended system text, project context, skills, current date, and working directory.
  5. User input is processed: extension commands may handle it directly; input hooks may transform it; /skill:name and prompt templates are expanded.
  6. Before model call: before_agent_start hooks can inject custom messages or modify the system prompt for this turn.
  7. Model receives: effective system prompt, conversation history, current user message, extension messages, and active tool schemas.
  8. Tool loop: model tool calls are intercepted by hooks, executed, appended as tool results, then sent back to the model until the assistant finishes.
Where different customization types enter the token stream
SourceWhere it enters
AGENTS.md / CLAUDE.mdAppended as project context in the system prompt.
SYSTEM.mdReplaces the built-in default system prompt.
APPEND_SYSTEM.mdAppended to the system prompt.
Prompt templateExpands the user message before the model call.
/skill:nameReads the full skill and injects it into the user message.
Available skills listAdvertised in the system prompt when the read tool is active.
ExtensionsCan alter tools, active tool list, system prompt, user input, messages, and tool calls.
Tool schemasSent separately to the model as callable tool definitions; they also consume input tokens.

6. Interactive usage basics

  • Type normal requests in the terminal editor.
  • Use @ to reference project files.
  • Use / to open commands, prompt templates, and skill commands.
  • Use !command to run shell and send output to the LLM.
  • Use !!command to run shell without sending output.
  • Common commands: /model, /settings, /resume, /new.

8. Extensions

Extensions are auto-discovered by file/folder convention.

LocationScope
~/.pi/agent/extensions/*.tsGlobal single-file extension
~/.pi/agent/extensions/*/index.tsGlobal folder extension
.pi/extensions/*.tsProject-local single-file extension
.pi/extensions/*/index.tsProject-local folder extension
Basic extension shape 
import type { ExtensionAPI } from "@earendil-works/pi-coding-agent";

export default function myExtension(pi: ExtensionAPI): void {
  pi.registerCommand("hello", {
    description: "Say hello",
    handler: async (_args, ctx) => {
      ctx.ui.notify("Hello from my extension", "info");
    },
  });
}
What extensions can do
  • Register slash commands with pi.registerCommand().
  • Register model-callable tools with pi.registerTool().
  • Register CLI flags with pi.registerFlag().
  • Listen to lifecycle/session/model/tool events with pi.on(...).
  • Block or modify tool calls.
  • Add status text, widgets, overlays, custom editors, and renderers.
  • Persist session state.
Security reminder: extensions are code and run with your system permissions. Only install or copy extensions you trust.

9. Plan mode extension

Project-local path:

.pi/extensions/plan-mode/index.ts
.pi/extensions/plan-mode/utils.ts
.pi/extensions/plan-mode/README.md
What it does
  1. Plan mode: restrict tools to read-only exploration.
  2. Bash allowlist: allow safe commands like ls, grep, find, git diff.
  3. Plan extraction: parse numbered steps from a Plan: section.
  4. Execution mode: restore normal tools and track progress.
  5. Completion markers: assistant marks steps with [DONE:n].
  6. UI progress: footer/status widget displays progress.

Commands: /plan, /todos, Ctrl+Alt+P.

10. Skills

Skills are self-contained instructions loaded on demand. They are useful for reusable workflows that do not need runtime TypeScript hooks.

.pi/skills/my-skill/SKILL.md

Use with:

/skill:my-skill
/skill:my-skill optional arguments here

Good for release workflows, framework conventions, documentation generation, debugging playbooks, helper scripts, and reference material.

11. Prompt templates

Prompt templates are Markdown files under .pi/prompts/. The filename becomes a slash command.

Example template 
---
description: Review staged git changes
argument-hint: "[focus]"

Review the staged changes with focus on: $ARGUMENTS

Check:
- correctness

15. Context files

Context files such as AGENTS.md give pi project-specific instructions: coding style, commands, test strategy, architecture notes, and repository conventions.

  • How to build and test the project.
  • Code style and naming conventions.
  • Important directories and architecture overview.
  • Safety rules, such as files never to edit.
  • Preferred workflows for commits, releases, or reviews.

16. Pi packages

Pi packages bundle extensions, skills, prompt templates, and themes so they can be shared via npm, git, URL, or local path.

pi install npm:@scope/package
pi install git:github.com/user/repo
pi install ./local-package

Use project-local install/settings when the package should belong to this repo.
