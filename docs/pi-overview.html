What is pi?

pi is a minimal terminal coding harness. It connects an LLM to a coding workspace and gives the model tools such as read, write, edit, and bash.

Instead of shipping every workflow as a built-in feature, pi expects users to adapt it with extensions, skills, prompt templates, themes, packages, settings, and context files.

Built-in and active tools

Pi has a distinction between registered tools and active tools.

ConceptMeaning
Registered toolsTools pi knows about. They may be built-in or provided by extensions.
Active toolsTools currently sent to the model as callable tool schemas.

Out of the box, pi normally starts with these active coding tools:

read, bash, edit, write

But pi also has additional built-in read-only exploration tools that can be activated by modes/extensions:

grep, find, ls
ToolTypeWhat it does
readDefault activeReads file contents. Supports text files and images. Large text output is truncated and can be continued with offset/limit.
bashDefault activeRuns shell commands in the current working directory. Useful for tests, git, search, scripts, and project commands.
editDefault activeMakes precise edits to existing files using exact text replacement. Best for surgical modifications.
writeDefault activeCreates or overwrites files. Best for new files or complete rewrites.
grepBuilt-in, not always activeSearches file contents for patterns. Faster/more structured than using bash grep and respects project ignore behavior.
findBuilt-in, not always activeFinds files by glob/pattern. Useful for codebase exploration without shell commands.
lsBuilt-in, not always activeLists directory contents. Useful for safe read-only exploration.
Example: the plan-mode extension activates grep, find, and ls even though default coding mode usually starts with only read, bash, edit, and write.

Default system prompt

This is pi's out-of-the-box default system prompt shape. The date, working directory, installed pi path, active tools, loaded context files, skills, and extensions can change the effective prompt at runtime.

Exact default prompt for this project/session 
You are an expert coding assistant operating inside pi, a coding agent harness. You help users by reading files, executing commands, editing code, and writing new files.

Available tools:
- read: Read file contents
- bash: Execute bash commands (ls, grep, find, etc.)
- edit: Make precise file edits with exact text replacement, including multiple disjoint edits in one call
- write: Create or overwrite files

In addition to the tools above, you may have access to other custom tools depending on the project.

Guidelines:
- Use bash for file operations like ls, rg, find
- Use read to examine files instead of cat or sed.
- Use edit for precise changes (edits[].oldText must match exactly)
- When changing multiple separate locations in one file, use one edit call with multiple entries in edits[] instead of multiple edit calls
- Each edits[].oldText is matched against the original file, not after earlier edits are applied. Do not emit overlapping or nested edits. Merge nearby changes into one edit.
- Keep edits[].oldText as small as possible while still being unique in the file. Do not pad with large unchanged regions.
- Use write only for new files or complete rewrites.
- Be concise in your responses
- Show file paths clearly when working with files

Prompt generation flow

Pi does not send only your typed text to the model. It assembles a structured model request from three streams:

StreamContainsExamples
A. Instruction streamThe effective system/developer prompt.default prompt or SYSTEM.md, APPEND_SYSTEM.md, AGENTS.md, available skills list, date/cwd, extension prompt changes.
B. Conversation streamMessages sent to the model.previous user/assistant messages, expanded current user message, extension custom messages, previous tool calls/results.
C. Tool interfaceCallable tool definitions/schemas.read, bash, edit, write, extension tools. These also consume input tokens.
Key distinction: AGENTS.md affects system prompt context, prompt templates affect your user message, skills are advertised in the system prompt and can be loaded into user context, and extensions can modify almost every stage.
1–4. Resource loading and base system prompt

On startup or /reload, pi discovers resources and keeps them available in memory for later: settings, packages, extensions, skills, prompt templates, themes, context files, SYSTEM.md, and APPEND_SYSTEM.md.

Prompt templates are discovered here so slash commands such as /review are ready. Skills are also discovered; their metadata can be advertised in the system prompt, while full skill content is usually loaded later when explicitly invoked or read.

The base system prompt starts with exactly one of these:

built-in default prompt
OR
SYSTEM.md

If SYSTEM.md exists, it totally replaces the built-in default prompt. After that, pi appends content in this order:

1. built-in default prompt OR SYSTEM.md
2. APPEND_SYSTEM.md / --append-system-prompt
3. AGENTS.md / CLAUDE.md project context
4. available skills list, if read tool is active
5. Current date
6. Current working directory

Append means extra system-level instructions added without replacing the base prompt. Use APPEND_SYSTEM.md for extra global/project rules; use AGENTS.md for repo/project guidance.

5–7. User input expansion and extension hooks

User input is processed before the model call:

  1. If it is an extension command such as /plan, the extension may handle it directly and no model call is required.
  2. input hooks can transform or handle the raw text/images.
  3. /skill:name args expands into a user message containing the full skill body plus args.
  4. Prompt templates such as /review foo expand into normal user-message text.
  5. before_agent_start hooks can add custom messages or modify the effective system prompt for this turn.

Important: /skill and prompt templates modify the conversation/user-message stream, not the base system prompt.

8. Final model request

The final request is structured, not one flat string. Conceptually:

{
  system: "effective system prompt",
  messages: [
    previous user messages,
    previous assistant messages,
    previous tool calls,
    previous tool results,
    current expanded user message,
    extension-injected custom messages
  ],
  tools: [
    read schema,
    bash schema,
    edit schema,
    write schema,
    extension tool schemas
  ]
}

All of these contribute to input tokens: system prompt, conversation history, current user message, custom messages, active tool schemas, and previous tool results. If the conversation grows too large, older history may be compacted/summarized depending on session behavior.

9. Tool loop

When the model wants to inspect or modify the workspace, it emits tool calls. Pi executes them and sends the result back to the model in another model call.

model asks for read(package.json)
→ pi emits tool_call hooks
→ extension may allow/block/modify
→ pi executes read
→ tool result is appended to conversation
→ model is called again with the result
→ repeat until assistant finishes

Extensions can intercept this loop to block dangerous bash commands, restrict tools in plan mode, ask for confirmation, log usage, modify inputs, or customize rendering.

Where each customization enters the token stream
SourceWhere it enters
AGENTS.md / CLAUDE.mdAppended as project context in the system prompt.
SYSTEM.mdReplaces the built-in default system prompt.
APPEND_SYSTEM.mdAppended to the system prompt after the base prompt.
Prompt templateExpands the user message before the model call.
/skill:nameReads the full skill and injects it into the user message.
Available skills listAdvertised in the system prompt when the read tool is active.
ExtensionsCan alter tools, active tool list, system prompt, user input, messages, and tool calls.

Extensions

Extensions are auto-discovered by file/folder convention.

LocationScope
~/.pi/agent/extensions/*.tsGlobal single-file extension
~/.pi/agent/extensions/*/index.tsGlobal folder extension
.pi/extensions/*.tsProject-local single-file extension
.pi/extensions/*/index.tsProject-local folder extension
Basic extension shape 
import type { ExtensionAPI } from "@earendil-works/pi-coding-agent";

export default function myExtension(pi: ExtensionAPI): void {
  pi.registerCommand("hello", {
    description: "Say hello",
    handler: async (_args, ctx) => {
      ctx.ui.notify("Hello from my extension", "info");
    },
  });
}
What extensions can do
  • Register slash commands with pi.registerCommand().
  • Register model-callable tools with pi.registerTool().
  • Register CLI flags with pi.registerFlag().
  • Listen to lifecycle/session/model/tool events with pi.on(...).
  • Block or modify tool calls.
  • Add status text, widgets, overlays, custom editors, and renderers.
  • Persist session state.
Security reminder: extensions are code and run with your system permissions. Only install or copy extensions you trust.

Extension events

Extensions listen to pi lifecycle events with pi.on(eventName, handler). Events let extensions observe or modify resources, session behavior, prompt/message context, tool calls, UI state, and provider requests.

pi.on("tool_call", async (event, ctx) => {
  if (event.toolName === "bash") {
    // inspect, mutate, or block the tool call
  }
});

Examples

Block dangerous bash:

pi.on("tool_call", async (event) => {
  if (event.toolName === "bash" && event.input.command.includes("rm -rf")) {
    return { block: true, reason: "Blocked dangerous command" };
  }
});

Inject hidden context before the agent starts:

pi.on("before_agent_start", async () => {
  return {
    message: {
      customType: "my-hidden-context",
      content: "Extra instructions for this turn",
      display: false,
    },
  };
});

Filter stale context messages:

pi.on("context", async (event) => {
  return {
    messages: event.messages.filter((m) => m.customType !== "stale-context"),
  };
});

Plan mode example study

Project-local path:

.pi/extensions/plan-mode/index.ts
.pi/extensions/plan-mode/utils.ts
.pi/extensions/plan-mode/README.md
What it does
  1. Plan mode: restrict tools to read-only exploration.
  2. Bash allowlist: allow safe commands like ls, grep, find, git diff.
  3. Plan extraction: parse numbered steps from a Plan: section.
  4. Execution mode: restore normal tools and track progress.
  5. Completion markers: assistant marks steps with [DONE:n].
  6. UI progress: footer/status widget displays progress.

Commands: /plan, /todos, Ctrl+Alt+P.

Study findings and gotchas
FindingWhy it matters
/plan toggles mode and resets todosThe example clears todoItems whenever plan mode is toggled, so toggling is not just a view change.
setActiveTools() replaces active toolsPlan mode uses hard-coded tool lists. It does not merge with tools activated by other extensions.
Other extension tools can be deactivatedIf another extension registered/activated custom tools, toggling plan mode can hide them from the model until restored manually.
NORMAL_MODE_TOOLS is hard-codedDisabling plan mode restores only read, bash, edit, write, not necessarily the user's previous active tool set.
grep, find, ls are built-in but not default activePi knows these tools, so plan mode can activate them even though normal coding mode usually starts with read, bash, edit, write.
questionnaire is not built-inIt comes from a separate example extension. If that extension is not loaded, the name is ignored by setActiveTools().
Registered tools vs active toolsPlan mode changes active tools. It does not create, delete, or unregister tools.

A more robust plan-mode implementation would save pi.getActiveTools() before enabling plan mode and restore that exact list when disabling it.
